## 0. 一句话结论

本次 L3 作业做了一个真实工作中能用的 App：**评审虾质量左移指标看板**。

它解决的是我当前 OKR 中的真实问题：评审虾质量左移不能只停留在“做了 AI 报告”“提了建议”，而要能持续记录试点需求、统计指标、保留证据链，并区分哪些结论已经确认，哪些还需要人工确认。

## 1. 原始需求

### 1.1 背景

我当前 6-7 月主线是评审虾质量左移量化体系，需要用指标体系和阶段报告证明质量左移与 SDD 提效贡献。实际推进时遇到几个问题：

| 问题 | 影响 |
|---|---|
| 指标口径容易散落在周报和文档里 | 难以持续跟踪 |
| AI 报告是否被采纳缺少记录 | 无法证明 AI 贡献 |
| PRD 打回和质量左移之间的因果关系容易模糊 | 对上汇报时证据不足 |
| 每个试点需求的证据和人工确认点没有统一位置 | 后续写月报/双月报告成本高 |
| 当前没有一个轻量工具帮助我快速看阶段数据 | 容易回到手工整理 |

### 1.2 原始需求描述

```text
我需要一个 PC 网页端 App，用来记录评审虾质量左移试点需求。

它需要支持：
1. 记录需求名称、负责人、阶段、风险等级。
2. 记录评审时长、是否使用 AI 报告、AI 建议是否被采纳、PRD 是否打回、左移问题数、交付评审是否通过。
3. 自动计算 5 个指标：评审时长、AI 报告采纳率、PRD 打回率、质量左移率、交付评审通过率。
4. 每条需求保留证据链和人工确认点。
5. 能筛选风险和阶段。
6. 能导出周同步 Markdown。
7. 能沉淀后续可复用的项目 Skills。
```

## 2. MVP 版本

### 2.1 本版做什么

| 模块 | MVP 功能 |
|---|---|
| 指标看板 | 展示 5 个核心指标，并基于记录自动计算 |
| 试点记录 | 内置样例数据，支持新增试点需求 |
| 筛选 | 支持按风险、阶段和关键词筛选 |
| 详情 | 展示单条需求的指标口径、证据链、人工确认点 |
| 人工确认 | 支持标记或取消人工确认 |
| 导出 | 将当前数据导出为周同步 Markdown 并复制到剪贴板 |
| 项目 Skills | 在界面展示 3 个可沉淀项目 Skill |

### 2.2 本版不做什么

```text
- 不做登录
- 不做云端数据库
- 不做权限系统
- 不做真实飞书项目 API 写入
- 不做多人协同
- 不做复杂图表库
- 不做历史版本对比
- 不公开任何内部链接或真实敏感数据
```

### 2.3 技术实现

| 项 | 方案 |
|---|---|
| 应用形态 | 静态 PC Web App |
| 页面技术 | HTML + CSS + JavaScript |
| 数据存储 | localStorage |
| 可视化 | 自定义指标卡和 SVG 小趋势线 |
| 运行方式 | 本地 HTTP server |
| 版本管理 | Git 仓库 |

## 4. Codex + Superpowers 执行方式

本项目按第三课方法执行：

```text
原始需求
→ MVP 范围
→ 开发计划
→ 任务拆解
→ Codex 实现
→ 验收检查
→ 修复
→ 项目 Skills 沉淀
```

本次使用到的 Superpowers 包括：

| Superpower | 用法 | 产出 |
|---|---|---|
| Project Skills | 把重复的评审记录、AI 采纳判断、证据链映射沉淀成 3 个 Skill | `project-skills/` |
| Browser 验收 | 用真实浏览器检查桌面端、手机宽度、交互和导出链路 | `docs/03_验收报告.md` |
| Git 版本控制 | 把可运行 App、报告和 Skills 做成一次可审阅提交 | 本地 Git 提交 |

### 4.1 任务拆解

| 任务 | Goal | 验收 |
|---|---|---|
| 数据模型 | 定义需求记录、指标、证据链、Skill 数据结构 | 样例数据能支撑指标计算 |
| 指标看板 | 展示 5 个指标 | 指标随数据变化 |
| 记录列表 | 展示和筛选试点需求 | 风险、阶段、搜索可用 |
| 详情面板 | 展示证据链和人工确认点 | 单击记录可查看 |
| 新增记录 | 表单新增本地记录 | localStorage 保存 |
| 导出同步 | 生成周同步 Markdown | 复制到剪贴板 |
| 项目 Skill | 沉淀可复用项目技能 | 3 个 SKILL.md 可校验 |

### 4.2 人工审核节点

| 节点 | 我审核什么 |
|---|---|
| MVP 定义 | 是否符合 L3 作业，也能服务真实 OKR |
| 指标口径 | 5 个指标是否和质量左移主线一致 |
| 数据字段 | 是否足够支撑周同步和双月报告 |
| UI 验收 | 是否像工作台，而不是营销页 |
| Skill 沉淀 | 是否真可复用，不是一次性说明 |

## 5. 项目 Skills

本项目沉淀 3 个项目 Skills：

| Skill | 解决的问题 |
|---|---|
| `review-metric-recording` | 把一次需求评审记录成可统计样本 |
| `ai-report-adoption-check` | 判断 AI 报告建议是否真实被采纳 |
| `left-shift-evidence-map` | 把质量左移结论映射到证据链 |

这些 Skills 满足第三课讲义中“值得沉淀”的条件：

1. 会重复出现。
2. AI 容易做错。
3. 有明确输入输出。
4. 可以被测试或验收。
5. 不是一次性项目说明。

## 7. 复盘

这次项目最大的收获不是做了一个页面，而是把第三课的方法用到真实工作：

- 先计划，再编码。
- 先 MVP，再扩展。
- 每个任务有边界和验收。
- 代码完成后要验收，不要直接相信生成结果。
- 真正有价值的是把项目中的重复动作沉淀成 Skills。

对我来说，这个 App 的价值是把“质量左移证明”从散落文档变成一个可持续记录的小工具；把“AI 写了报告”变成“AI 建议是否被采纳、是否前置发现问题、是否能形成证据链”的可检查流程。
